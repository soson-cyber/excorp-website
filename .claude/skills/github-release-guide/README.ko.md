# github-release-guide

[English](./README.md) · **한국어**

GitHub 저장소를 안전하게 공개하려면 릴리스 버튼만 누르면 되는 것이 아닙니다. README와 설치 방법, 라이선스,
버전, 태그(tag), 저장소 설정이 서로 맞아야 하고, 공개된 뒤에는 되돌리기 어려운 변경도 있습니다.

`github-release-guide`는 지금 무엇이 준비됐고, 무엇을 더 확인해야 하며, 다음에 어떤 결정을 내려야 하는지
알기 쉽게 정리해 줍니다. 준비가 끝난 뒤에는 사용자가 승인한 변경만 한 단계씩 실행하도록 안내합니다.
릴리스를 결정하는 권한은 모든 과정에서 사용자에게 남아 있습니다.

> **실행 환경 지원 상태:** Claude Code와 Codex는 정답을 제거한 가상 예제에서 같은 핵심 행동을 보였고,
> 일회용 검증 저장소의 실제 첫 공개 E2E도 통과했습니다. 두 실행 환경 모두 완전한 `v0.5.0` 고정 버전
> 프로젝트 패키지를 발견했으며, 최종 공개 문구 검증도 통과했습니다. 기록된 검증 범위에서 현재 표시는
> `Supported`입니다.
>
> **성숙도: Stable** — Claude Code/Codex 핵심 행동 일치(보호 설정 예제 PT1~PT3 포함), 첫 공개
> E2E, Guided tag-ruleset 적용·검증 E2E(2026-07-17), 고정 버전 설치·발견 검증을 근거로
> 합니다. 실행 환경 지원(`Supported`)과 성숙도(`Stable`)는 별개의 표시입니다.

## 어떻게 진행할지 선택하세요

| 방식 | 진행 내용 |
| --- | --- |
| `Assess`(점검) | 확인할 수 있는 정보만 읽고 `Ready`, `Needs attention`, `Blocked` 중 하나로 상태를 알려줍니다. 그 이유와 가장 안전한 다음 행동 하나를 설명하며 저장소는 변경하지 않습니다. |
| `Guided`(단계별 진행) | 점검이 끝나면 변경할 내용 하나를 먼저 보여줍니다. 현재 상태를 다시 확인하고 사용자의 직접 승인을 받은 뒤, 승인된 작업만 실행하고 실제 결과를 검증합니다. |

무엇부터 해야 할지 모르겠다면 `Assess`로 시작하세요. 계획 전체에 동의했더라도 그것만으로 저장소 변경이
승인되지는 않습니다. 실제 변경마다 별도의 확인을 거칩니다.

## 어떤 릴리스를 준비하는지 선택하세요

### `first-public` — 비공개 저장소를 처음 공개할 때

이미 github.com에 있는 비공개 저장소를 처음 공개할 때 사용합니다. 공개 전에 README와 설치 방법, 라이선스
결정, 첫 버전과 태그, 민감 정보 노출 가능성, 저장소 설정, 릴리스 노트를 확인합니다.

한 번 공개된 내용은 누군가 복제하거나 저장한 뒤에 완전히 회수할 수 없습니다. 또한 자동 검사는 모든 위험을
찾아낸다고 보장할 수 없습니다. 그래서 저장소를 공개하기 직전에 이 한계를 다시 설명하고, 공개 전환에 대한
사용자의 직접 승인을 별도로 확인합니다.

### `version-release` — 공개 후 새 버전을 릴리스할 때마다

이미 공개된 github.com 저장소에서 새 버전을 준비할 때마다 사용합니다. 첫 공개 뒤 첫 번째 버전 릴리스 한
번으로 제한되지 않습니다. 목표 버전, 기준이 되는 버전 파일, CHANGELOG, 릴리스 노트, 대상 커밋(commit),
기존 태그 충돌, 설치 및 호환성 근거를 확인합니다. 파일 수정, 커밋, 원격 저장소 전송(push), 태그 생성,
GitHub Release 공개는 각각 별도의 승인 단계로 진행합니다.

태그를 만들기 전에는 릴리스 보호 상태도 확인합니다 — 기본 브랜치와 릴리스 태그가 실제 릴리스 방식에
맞게 보호되어 있는지 봅니다. 사용자가 고정된 태그(pinned tag)에 의존하는데 보호가 없다면 현황만
알리는 데 그치지 않습니다. Guided에서는 권장 설정을 적용하고 결과까지 검증하겠다고 제안하며, 각 적용은
별도의 승인 단계로 진행합니다. 거절하면 저장소를 바꾸지 않고 수용한 위험(accepted risk)으로 기록합니다.

## 공개 전에 무엇을 확인하나요?

Assess는 현재 접근할 수 있는 자료를 확인하고, 확인하지 못한 항목은 그대로 알려줍니다. 아래 점검은 공개
위험을 줄이는 데 도움을 주지만, 모든 위험을 자동으로 차단하거나 전문 보안 감사를 대신하지는 않습니다.
두 릴리스 유형 모두에 적용되며, 저장소 전체와 이력이 처음 노출되는 `first-public`에서 특히 중요합니다.

| 점검 범주 | 확인하는 내용 | 중요한 한계 |
| --- | --- | --- |
| 인증 정보와 민감 정보 | 추적 중인 파일에 API 키, 토큰, 비밀번호와 비슷한 값이 있는지 확인 | 실제 인증 정보 값을 요구하거나 출력하지 않음 |
| Git 이력 | 현재 파일에서 삭제됐지만 이전 커밋에 남아 있을 수 있는 민감 정보 | 가능한 범위에서 확인하며 전문적인 전체 이력 조사를 대신하지 않음 |
| 개인 환경 흔적 | 개인 컴퓨터 경로, 사용자명, 이메일, 계정·조직 식별자 | 식별자가 있다는 이유만으로 비밀로 판정하지 않고 공개 의도를 확인 |
| 내부 연결 정보 | localhost, 내부 서버 주소, 사설 네트워크 이름, 사내 도메인 | 존재만으로 릴리스를 막지 않고 공개 대상과 의도를 확인 |
| 생성 파일과 메타데이터 | 빌드 결과물, 압축 파일, PDF·Office 파일, 이미지, 스크린샷과 포함된 메타데이터 | 현재 실행 환경에서 읽을 수 있는 형식과 내용만 확인 가능 |
| 설정과 자동화 파일 | 환경 변수, 설정, CI와 배포 파일이 민감한 값을 포함하거나 참조하는지 확인 | 실제 값을 노출하지 않고 관련 파일과 위험만 확인 |
| GitHub 보안 설정 | Secret scanning, push protection, 취약점 경고와 미해결 경고 상태 | 요금제, 권한과 조직 정책에 따라 확인하지 못할 수 있음 |

점검 결과가 깨끗하다는 말은 확인한 범위에서 문제를 발견하지 못했다는 뜻입니다. 저장소에 민감 정보나 보안
위험이 전혀 없다는 보장은 아닙니다. 확인할 수 없는 항목은 통과한 것으로 처리하지 않고 점검 결과에
확인하지 못한 항목으로 분명하게 표시합니다.

## 시작 전에 준비하면 좋은 정보

모든 정보를 미리 알고 있을 필요는 없습니다. 모르는 항목은 추측하지 않고 점검 결과에 그대로 표시합니다.
가능하다면 다음 정보를 준비해 주세요.

- 정확한 GitHub 소유자와 저장소 이름
- 현재 저장소가 비공개인지 공개인지
- 공개하려는 첫 버전 또는 이번에 배포할 버전
- 저장소의 브랜치(branch) 및 릴리스 규칙
- README, 설치 문서, CHANGELOG, 라이선스 결정, 릴리스 노트 초안
- 최근 테스트, 빌드, 복제(clone) 또는 설치 결과
- 저장소 변경을 승인할 권한이 있는 사람

비밀번호나 접근 토큰(access token) 같은 인증 정보의 실제 값은 스킬에 제공하지 마세요.

## 반드시 지키는 안전 원칙

- 중요한 정보가 확인되지 않으면 중단 사유로 분명하게 알려주며, 성공한 것으로 추측하지 않습니다.
- 저장소에 영향을 주는 변경은 매번 내용과 영향을 먼저 보여주고 별도로 승인받습니다.
- 변경 직전에 브랜치, 태그, 공개 상태 등 중요한 조건을 다시 확인합니다. 상태가 달라졌다면 이전 승인은
  무효가 되고 새 내용을 다시 보여줍니다.
- 저장소 공개 전환은 설정 변경, 태그 생성, GitHub Release 공개와 한꺼번에 처리하지 않습니다.
- 작업 중 오류가 발생하거나 인증 정보 노출이 의심되면 즉시 중단합니다. 이 경우 단순히 되돌리는 대신
  보안 사고 대응 절차에 따라 처리합니다.
- 강제 전송(force-push)이나 공개된 커밋 기록 다시 쓰기(history rewrite)는 실행하지 않습니다.

## 사용 예시

### Assess

```text
github-release-guide를 Assess 방식으로 사용해서 이 비공개 저장소를 공개할 준비가 됐는지 점검해 줘.
```

```text
github-release-guide를 Assess 방식으로 사용해서 이 공개 저장소의 이번 버전 릴리스를 점검해 줘.
```

### Guided

```text
github-release-guide를 Guided 방식으로 사용해서 이 비공개 저장소의 첫 공개를 준비해 줘.
먼저 Assess 결과를 보여주고, 준비가 완료됐으면 첫 변경의 내용과 영향만 미리 보여줘.
내가 그 작업을 직접 승인하기 전에는 저장소나 공개 상태를 변경하지 마.
```

`Guided` 방식을 요청해도 릴리스 전체가 한꺼번에 승인되는 것은 아닙니다. 먼저 Assess로 준비 상태를
확인하고, 이후 변경 작업도 하나씩 다시 승인받습니다.

## 지원 범위

V1은 github.com 전용입니다. 이미 존재하는 비공개 저장소를 처음 공개 상태로 전환할 때 사용하며,
공개된 뒤에는 새로운 버전을 릴리스할 때마다 다시 사용할 수 있습니다. 저장소 생성, Git 초기화,
패키지 저장소 공개(package registry publish), 바이너리 서명, 클라우드 배포, 보안 감사, 강제 전송,
커밋 기록 다시 쓰기는 수행하지 않습니다.

설치 폴더에는 이 README, 에이전트가 읽는 실행 지침인 `SKILL.md`, Codex 화면에 표시되는 정보,
공통 점검 규칙과 릴리스 유형별 참고 규칙이 들어 있습니다. 검증에 사용한 가상 예제와 다이어그램은
설치 폴더가 아니라 Skillstead 저장소에서 제공합니다.
