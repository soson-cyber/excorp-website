# EX 디자인·UX·접근성 검토

검토·제안 중심 스킬이다. 구현은 frontend-builder/ex-frontend-implementation이 한다. 모든 권고에 **무엇·어디(파일/라인)·왜·어떻게**를 붙이고, 우선순위(High/Med/Low)를 매긴다.

## 1. DESIGN.md 정합성 (1순위)
토큰 일탈(하드코딩 색)·키트 클래스 미사용·임의 인라인 스타일·톤 교차(`--ink/--surface/--white`) 위반을 잡는다. 단일 기준은 `DESIGN.md §0~§9`.

## 2. 반응형 (항상 두 모드)
- 전역 `word-break: keep-all` 전제. 수동 `<br/>`가 모바일(360~390px)에서 만드는 **고아행·의미단위 깨짐**을 지적한다.
- 권고 규칙: **본문=`<br/>` 제거 + text-wrap**, **제목=데스크톱 한정 `hidden sm:block`(twocol은 `hidden lg:block`) + `{" "}` 공백 보존.**
- `.h2` 모바일 line-height(대형 타이포 2~3줄 시 빡빡함), clamp 하한, 중앙정렬 장문(모바일 가독 저하 → 좌측 권고)을 본다.

## 3. 접근성 (WCAG)
- 대비 AALAST: 본문급 텍스트 알파 0.6은 0.7로 상향 검토. 장식 caption(10~11px)은 예외.
- 포커스 가시성(`:focus-visible`/`.focus-on-dark`), 키보드 내비, 모바일 시트의 포커스 트랩·Esc·스크롤락, 탭 타깃 44px.
- `prefers-reduced-motion` 정적 경로, 장식 요소 `aria-hidden`, 의미 있는 `alt`.

## 4. 정보 위계·스캔성·CTA
타이포 위계(eyebrow/`.h2`/`.lead`), 여백 리듬, CTA 라벨 일관성(같은 행동에 표현 분산 금지), 카드 내부 정렬.

## 실제 렌더 확인
프리뷰가 필요하면 qa-verifier에 데스크톱(1320)+모바일(375) 스크린샷을 요청한다. 헤드리스 스크린샷은 상단만 신뢰 가능 → 중간 섹션은 eval 스크롤 후 캡처하도록 명시.

## 출력 형식
```
## 검토 결과 (대상: ...)
### 🔴 High
1. [무엇] — 파일:라인 — 왜 — 어떻게(마크업/토큰 예시)
### 🟡 Med  ### 🟢 Low
### 반응형 / 접근성 별도 섹션
```
직접 수정하지 않는다. frontend-builder에 우선순위·파일·예시를 넘긴다.
