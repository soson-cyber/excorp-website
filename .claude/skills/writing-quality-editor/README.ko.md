# writing-quality-editor

[English](./README.md) · **한국어**

사용자가 읽는 글을 처음부터 잘 쓰거나 숙련된 편집자가 다듬은 것처럼 자연스럽고 명료하게 고칩니다.
문장을 새로 만들더라도 제공된 사실과 근거의 범위는 바꾸지 않습니다.

README, 온보딩 문서, 릴리스 노트, 사용 설명서, 앱 UI, 오류 메시지와 갤러리 문구를 처음 작성하거나
검토·수정할 때 사용할 수 있습니다. 신뢰할 수 있는 작성 자료에서 곧바로 새 문서를 쓰고, 같은 언어
안에서 기존 문장을 고치거나, 영어와 한국어 사이에서 대상 독자에게 맞는 글로 다시 쓸 수 있습니다.

## 네 가지 mode

| Mode | 용도 | 파일 변경 |
| --- | --- | --- |
| `Compose` | 제공된 사실·근거·제약과 독자 목표로 새 문서 작성 | 요청받은 초안이나 파일만 생성 |
| `Assess` | 의미 보존, 명료성, 문체, 구조, 현지화 문제 진단 | 변경하지 않음 |
| `Revise` | 같은 언어 안에서 문장 품질 개선 | 요청받은 범위만 변경 |
| `Adapt` | 영어와 한국어 사이에서 대상 언어로 처음 작성한 것처럼 자연스럽게 재구성 | 요청받은 범위만 변경 |

`Adapt`는 단어를 일대일로 옮기는 번역이 아닙니다. 독자와 문서 종류에 맞게 문장을 나누거나 합치고,
정보 순서와 관용 표현, 설명 밀도를 조정할 수 있습니다. 대신 사실과 주장, 의도, 작성자 목소리, 조건,
수치, 식별자, 예외, 제약, 위험, 승인과 다음 행동은 그대로 보존합니다.

`Compose`는 초안을 만든 뒤 다시 다듬는 이중 작업을 줄입니다. 처음부터 독자와 문서 목적에 맞게 쓰되,
문장만 새로 만들고 사실은 새로 만들지 않습니다. 제공되지 않은 기능·근거·호환성·수치·운영 결정은
그럴듯하게 채우지 않고 필요한 판단으로 남깁니다.

사용자가 자료를 직접 주지 않아도 공개 자료를 조사할 수 있다면 `Compose`가 먼저 근거를 수집할 수
있습니다. 검토한 출처와 근거 기준일을 기록하고 주요 주장에 출처를 표시하며, 측정된 사실과 출처의 주장,
작성자의 종합 판단을 구분합니다. 공개되어 있다는 이유만으로 자료를 신뢰하지 않습니다.

## 반드시 보존하는 것

- 사실과 이를 뒷받침하는 근거의 범위
- 작성자의 의도와 목소리
- 명령, 경로, URL, 오류 코드, 제품명, 버전과 기타 식별자
- 조건, 예외, 제약, 불확실성, 위험, 승인과 되돌리기의 의미
- 기준 문서와 이를 따라가는 문서의 관계, 링크

두 언어에서 안전하게 대응되는 표현이 없거나 원문이 모호하면, 매끄러운 문장 속에 임의의 해석을 숨기지
않고 `needs-human`으로 구분해 보여줍니다.

## 여기서 말하는 자연스러운 글

원문 언어의 어순이 그대로 비치는 문장, 설명되지 않은 내부 은유, 내용 없는 도입, 반복되는 요약,
기계적인 대칭, 똑같은 문장 호흡, 과도한 확신, 독자가 목적을 알기 전에 나오는 기술 설명을 찾아
문맥에 맞게 고칩니다. 특정 단어를 일괄 금지하거나, 필요한 기술 용어를 없애거나, 이미 자연스러운
문장을 굳이 다르게 쓰지는 않습니다.

AI 탐지기를 속이기 위한 도구는 아닙니다. 작성 주체를 알아볼 수 없다고 보장하지 않으며, 작성 주체나
출처를 숨기거나 가짜 경험을 넣거나 무작위로 낯선 표현을 섞지 않습니다.

## 검증 범위

- 특정 언어에 묶이지 않는 글쓰기 절차로 설계
- 첫 릴리스에서 검증하는 언어 조합: 영어↔한국어(한국어 결과는 `ko-KR`)
- 가상 검증 자료가 있는 문서 종류: README, 온보딩 문서, 릴리스 노트, 사용 설명서, 앱 UI,
  오류 메시지, 갤러리 문구
- 성숙도: Beta. 첫 릴리스의 지원 범위를 좁게 유지하면서 문서 종류와 언어별 근거를 쌓는 단계

위 항목은 실제 근거가 있는 범위만 표시한 것입니다. 다른 언어나 문서 종류에도 도움이 될 수 있지만,
검증된 지원 범위라고 주장하지 않습니다.

## 사용 예시

실행 환경에 관계없이 스킬을 확실히 선택하려면 이름을 밝히고 원하는 결과를 자연스럽게 말합니다.
mode까지 지정할 필요는 없습니다.

```text
writing-quality-editor를 사용해 아래 문서를 자연스럽게 다듬어 줘. 원문의 핵심 사실, 조건과 요구 사항은
그대로 유지해 줘.
```

설치된 에이전트가 요청의 의도를 파악해 스킬을 선택할 수 있는 환경에서는 평소처럼 요청해도 됩니다.

```text
이 제품 설명 자료를 바탕으로 새 README를 작성해 줘. 사용자 가치를 먼저 설명하고, 자료에 없는 기능이나
지원 환경은 추정하지 마.
```

```text
이 README를 검토해 줘. 내부 구조를 모르는 첫 사용자가 제품 가치와 다음 행동을 이해할 수
있는지 알려 줘. 아직 문장은 고치지 마.
```

```text
현재 기업의 AI 전환 도입 추세를 조사해 brief로 작성해 줘. 주요 주장에 출처를 달고, 근거 기준일을
밝혀 줘. 측정된 도입 현황과 공급업체 발표, 작성자의 추론도 구분해 줘.
```

```text
이 영어 온보딩 안내서를 한국어 독자가 자연스럽게 읽을 수 있도록 다시 써 줘. 명령과 경로는
번역하지 말고 모든 전제 조건, 경고와 복구 절차를 보존해 줘.
```

작업 범위를 명확히 고정해야 할 때만 mode를 직접 지정합니다.

```text
writing-quality-editor를 Assess mode로 사용해 이 릴리스 노트를 검토해 줘. 문장은 고치지 마.
```

저장소 자체의 절차가 brief 분류, 파일 위치, index 또는 승인을 관리한다면 그 절차를 먼저 따릅니다.
`writing-quality-editor`는 그렇게 정해진 문서 계약 안에서 문장 품질을 높일 수 있지만, 저장소의
작업 절차나 승인 규칙을 대신하지 않습니다.
